in this proposal. The algorithms selected are a result of balancing a few things:
\begin{itemdescr}
\begin{itemize}
      \item Include a rich enough set of algorithms for the library to be useful.
      \item Include algorithms with well-defined functionality and agreed-upon algorithmic description.
      \item Don't include so many that the proposal will get bogged down for years and years.
\end{itemize}
\end{itemdescr}

\subsection{Ordered Vertex Edges Concept}
The \lstinline{ordered_vertex_edges} concept requires that the outgoing edges of each vertex
are sorted in ascending order of \lstinline{target_id}. This is needed by algorithms such as
\lstinline{triangle_count} that rely on sorted adjacency lists for efficient intersection.
This is a semantic requirement that is not enforced by the graph container interface, but it 
is documented as a requirement for algorithms that need it.

\begin{lstlisting}
// For exposition only
template <class G>
concept ordered_vertex_edges = adjacency_list<G> &&
  requires(G& g, vertex_t<G> u) {
    requires forward_iterator<decltype(ranges::begin(out_edges(g, u)))>;
  };
\end{lstlisting}

\subsection{Vertex Property Map Concept}
Algorithms that maintain per-vertex state (distances, predecessors, component labels, etc.)
require a container indexed by vertex ID.  For index-based graphs a \tcode{vector} is natural;
for mapped (key-based) graphs an \tcode{unordered_map} is required.
The \tcode{vertex_property_map<G,T>} type alias (defined in [D3130]) selects the appropriate
container automatically.

Two additional utilities are defined here for use in algorithm \tcode{requires} clauses
and algorithm bodies.

\subsubsection{\tcode{vertex_property_map_value_t}}
\tcode{vertex_property_map_value_t<C>} extracts the stored value type from any vertex
property map container:
\begin{itemize}
  \item \tcode{vector<T>} $\Rightarrow$ \tcode{T} (via \tcode{value_type})
  \item \tcode{unordered_map<K,T>} $\Rightarrow$ \tcode{T} (via \tcode{mapped_type})
\end{itemize}

\begin{lstlisting}
// For exposition only
template <class Container>
using vertex_property_map_value_t = /* value_type or mapped_type of Container */;
\end{lstlisting}

\subsubsection{\tcode{vertex_property_map_for} Concept}
\tcode{vertex_property_map_for<M,G>} constrains algorithm parameters such as
\tcode{Distances} and \tcode{Predecessors}.  It requires subscript access by
\tcode{vertex_id_t<G>}:

\begin{lstlisting}
// For exposition only
template <class M, class G>
concept vertex_property_map_for =
    requires(M& m, const vertex_id_t<G>& uid) {
      { m[uid] } -> convertible_to<vertex_property_map_value_t<M>>;
    };
\end{lstlisting}

This concept is satisfied by:
\begin{itemize}
  \item \tcode{vector<T>} for index graphs (\tcode{vertex_id_t<G>} is integral)
  \item \tcode{unordered_map<vertex_id_t<G>,T>} for mapped graphs
  \item Any other subscriptable container keyed on \tcode{vertex_id_t<G>}
\end{itemize}

See [D3130] for the definitions of \tcode{vertex_property_map<G,T>},
\tcode{make_vertex_property_map}, \tcode{vertex_property_map_contains},
\tcode{vertex_property_map_get}, and the concepts \tcode{mapped_adjacency_list<G>},
\tcode{hashable_vertex_id<G>}, and \tcode{mapped_vertex_range<G>}.

\subsection{Visitor Concepts and Classes}
Visitors are optional member functions on a user-defined class that are called during the execution of an algorithm. 
Each algorithm has its own set of visitor events that it supports, and each event function must match the visitor concepts 
shown in this section.

The visitor events mimic those used in the Boost Graph Library.

\subsubsection{Vertex Visitor Concepts}
{\small
      \lstinputlisting{D3128_Algorithms/src/visitor_vertex.hpp}
}
Each vertex event concept exists in two variants.
The \textit{descriptor} variant (e.g.\ \lstinline{has_on_initialize_vertex}) requires an event
function whose second argument is \lstinline{const vertex_t<G>&}.
The \textit{ID} variant (e.g.\ \lstinline{has_on_initialize_vertex_id}) requires an event
function whose second argument is \lstinline{const vertex_id_t<G>&}.
The ID variant is useful when the visitor only needs the vertex identifier rather than the full
vertex descriptor.  In both variants the first argument is \lstinline{const G&}.

The vertex events are called under the following conditions.
\begin{itemize}
      \item \lstinline{on_initialize_vertex(g, vdesc)} (or \lstinline{on_initialize_vertex(g, uid)})
            is called once for each vertex before the algorithm is run.
      \item \lstinline{on_discover_vertex(g, vdesc)} (or \lstinline{on_discover_vertex(g, uid)})
            is called once for each source vertex passed to the algorithm.
      \item \lstinline{on_examine_vertex(g, vdesc)} (or \lstinline{on_examine_vertex(g, uid)})
            is called for a vertex before any of its outgoing edges are examined.  It is possible
            that it will be called multiple times for the same vertex if paths are found to it
            from other vertices with a shorter distance.
      \item \lstinline{on_start_vertex(g, vdesc)} (or \lstinline{on_start_vertex(g, uid)})
            is called for a vertex that is being examined.
      \item \lstinline{on_finish_vertex(g, vdesc)} (or \lstinline{on_finish_vertex(g, uid)})
            is called for a vertex that is being examined, after all its outgoing edges have been
            examined.
\end{itemize}

\subsubsection{Edge Visitor Concepts}
{\small
      \lstinputlisting{D3128_Algorithms/src/visitor_edge.hpp}
}
The edge events are called under the following conditions.
\begin{itemize}
      \item \lstinline{on_examine_edge(edesc)} is called for edge of the source vertex that is
            being examined.
      \item \lstinline{on_edge_relaxed(edesc)} is called when the distance to the target vertex of the edge
            is relaxed, or decreased.
      \item \lstinline{on_edge_not_relaxed(edesc)} is called when the distance to the target vertex of the edge
            is not relaxed, or not decreased.
      \item \lstinline{on_edge_minimized(edesc)} is called when the distance to the target vertex of the edge
            is minimized, or decreased to the minimum value.
      \item \lstinline{on_edge_not_minimized(edesc)} is called when the distance to the target vertex of the edge
            is not minimized, or not decreased to the minimum value.
      \item \lstinline{on_tree_edge(edesc)} is called when the edge is added to the tree.
      \item \lstinline{on_back_edge(edesc)} is called when the edge is a back edge.
      \item \lstinline{on_forward_or_cross_edge(edesc)} is called when the edge is a forward or cross edge.
      \item \lstinline{on_finish_edge(edesc)} is called when the edge is finished being examined.
\end{itemize}

The Bellman-Ford algorithm supports the use of negative edge weights, at cost in performance. 
Because of its complexity, it can only be used for small graphs.

that is reachable from the source, then there is no cheapest path: any path that has a point 
on the negative cycle can be made cheaper by one more walk around the negative cycle. In such 
a case, the Bellman–Ford algorithm can detect and report the negative cycle." 
\textbf{[Wikipedia, Bellman–Ford algorithm]}

get the vertex ids of the negative weight cycle.

\subsection{Triangle Counting}
Compute the number of triangles in a graph. A triangle is a set of three vertices $\{u, v, w\}$
where each pair is connected by an edge. Two functions are provided:
\tcode{triangle_count} counts triangles (3-cliques) in undirected graphs, and
\tcode{directed_triangle_count} counts directed 3-cycles in directed graphs.

Both algorithms use a merge-based set intersection approach on sorted adjacency lists,
which is more efficient than nested-loop or hash-based methods for sparse graphs.
The algorithms require the graph to satisfy both \tcode{adjacency_list<G>} and
\tcode{ordered_vertex_edges<G>}.

                        \item Perform a merge-based intersection of the two lists
                              (analogous to \tcode{std::set_intersection}).
                        \item For each common neighbor $w$ where \tcode{vertex_id(g, w)} $>$ \tcode{vertex_id(g, v)},
                              increment the triangle count.
                  \end{enumerate}
                  \item The ordering constraint
                        \tcode{vertex_id(g, u)} $<$ \tcode{vertex_id(g, v)} $<$ \tcode{vertex_id(g, w)}
                        guarantees each triangle is counted exactly once.
            \end{itemize}

            \end{itemize}

            \begin{itemize}
                  \item Edge weights, if present, are ignored.
                  \item Self-loops are ignored (a vertex cannot form a triangle with itself).
                  \item Multi-edges: each parallel edge contributes to the triangle count separately.
                  \item Works correctly on both connected and disconnected graphs.
                  \item The \tcode{ordered_vertex_edges} requirement enables $\mathcal{O}(d)$ set intersection

                        performance on high-degree vertices. Graph types using \tcode{std::set} or
                        \tcode{std::map} edge containers (e.g.\ \tcode{vos}, \tcode{uos}, \tcode{dos},

            \end{itemize}

\subsection{Label Propagation}
Propagate vertex labels for community detection by majority voting among neighbours.
Each iteration shuffles the vertex processing order, then sets every vertex's label to the most
popular label among its neighbours. Ties are broken randomly using the supplied random-number
generator. The algorithm iterates until no label changes (convergence) or until \tcode{max_iters}
iterations have been performed. $\mathcal{O}(M)$ complexity, where $M = |E|$, is based on one
iteration (each edge is examined once); the number of iterations required for convergence is
typically small relative to graph size.

A second overload accepts an \tcode{empty_label} sentinel for semi-supervised propagation:
vertices whose label equals \tcode{empty_label} are treated as unlabelled --- they do not vote
and are not counted in neighbour tallies, but can acquire a label from a labelled neighbour.
Vertices in components with no labelled vertex retain \tcode{empty_label} after convergence.

\section{Components}
\subsection{Articulation Points}
Find articulation points (cut vertices) of a graph. An articulation point is a vertex whose removal,
along with its incident edges, disconnects the graph into two or more connected components.
The algorithm uses an iterative Hopcroft-Tarjan DFS, tracking discovery times (\tcode{disc[v]}) and
low-link values (\tcode{low[v]}), defined as the minimum discovery time reachable from the subtree
rooted at $v$ via back-edges. A vertex $u$ is an articulation point under one of two rules:
\begin{itemize}
      \item \textbf{Root rule:} $u$ is the root of a DFS tree and has two or more DFS children.
      \item \textbf{Non-root rule:} $u$ is not a root and has a child $v$ with \tcode{low[v]} $\geq$ \tcode{disc[u]}.
\end{itemize}

                        \lstinline{compare} operator, when provided, is a valid comparison operation on two edge values
                        of type \lstinline{range_value_t<EL>::value_type} which returns a bool.
            \end{itemize}
      \pnum\effects 
            \begin{itemize}
                  \item
                        The input edge list \lstinline{e} is sorted by edge weight (or by \lstinline{compare}
                        when provided).
                  \item
                        Edgelist \lstinline{t} contains edges representing a spanning tree or forest, which
                        minimize the comparison operator.  When \lstinline{compare} is \lstinline{<},
                        \lstinline{t} represents a minimum weight spanning tree.
            \end{itemize}
      %\pnum\result
      \pnum\returns A \lstinline{pair<EV, size_t>} containing the total weight of the spanning tree or
            forest and the number of connected components.
      %\pnum\throws
      \pnum\complexity
            \begin{itemize}
                  \item \textbf{Time:} $\mathcal{O}(|E|\log{|E|})$.
                  \item \textbf{Space:} $\mathcal{O}(|V|)$ auxiliary --- union-find only; no edge copy.
            \end{itemize}
      \pnum\remarks The input edge list is modified (sorted).  If the caller needs to
            preserve the original order, use \lstinline{kruskal} instead.
      %\pnum\errors
\end{itemdescr}
